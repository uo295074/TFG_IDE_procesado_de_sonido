 Tecnologías a utilizar

1.	Entorno de desarrollo -> Puedo utilizar Visual Studio Code, añadiendo extensiones de C++ o Python.

    1.1. Lenguaje de programación decidido -> Para el desarrollo del proyecto he decidido utilizar C++ como lenguaje principal junto con el framework JUCE. La elección de C++ es fundamental para garantizar el rendimiento en tiempo real y la baja latencia que requiere el procesamiento de audio digital en Linux. Por su parte, JUCE se ha seleccionado por ser el estándar actual de la industria; ofrece una solución integral que abstrae la complejidad de la comunicación con los sistemas de audio nativos (ALSA/JACK) y proporciona un potente sistema de widgets gráficos tanto para la interfaz del IDE como para los plugins resultantes. Esta combinación permite agilizar el desarrollo al unificar la lógica de procesamiento (DSP) y la interfaz gráfica (GUI) bajo una misma arquitectura, asegurando además la disponibilidad de documentación técnica robusta para la implementación del sistema de generación automática de código.

2.	Framework de interfaz gráfica -> JUCE

3.	Motor de Generación/ Plantillas (Scaffolding) -> El flujo de trabajo comienza definiendo un Modelo de Datos que actúa como el plano maestro del sistema. Cuando el usuario interactúa con la interfaz gráfica, tu programa crea un objeto interno (por ejemplo, una clase PluginModel) que almacena la lista secuencial de los módulos que compondrán el plugin, tales como la entrada estéreo, el control de ganancia, el algoritmo de reverb y la salida final. A continuación, interviene el Gestor de Secciones, un componente crucial debido a la arquitectura de JUCE, que exige que el código resida en ubicaciones muy específicas y no de forma lineal. Este generador se encarga de separar cada módulo en sus "trozos" lógicos correspondientes: define qué librerías incluir en los Headers, declara las variables privadas necesarias, establece el código de inicialización en el Setup e inserta la lógica de procesamiento de señal dentro del bucle de audio o Process. La fase de construcción concluye con el Ensamblador. En este paso, el IDE recorre la lista de módulos definida en el modelo, extrae los fragmentos de código preparados por el gestor para cada sección específica e inyecta estos bloques en la plantilla maestra de JUCE, generando así el archivo de código fuente completo y estructurado. Este enfoque modular es muy beneficioso para la calificación del TFG por varias razones de ingeniería. Primero, garantiza la escalabilidad, ya que añadir un nuevo módulo (como un Chorus) no requiere reescribir el generador. Segundo, permite una validación previa, evitando errores lógicos como conectar una salida mono a una entrada estéreo antes de escribir una sola línea de código. Finalmente, enriquece la memoria técnica, permitiéndote dedicar capítulos profundos a explicar conceptos avanzados como algoritmos de inyección de dependencias o motores de ensamblaje.

5.	Compilador e Instalador Automatizado -> Compilador: GCC (Compilador nativo de linux, viene en el paquete build-essential). Sistema de Construccion: CMake (JUCE tiene soporte oficial para CMAKE. El IDE solo tendrá que generar un archivo de texto llamado CMakeLists.txt junto con el codigo. Luego el IDE ejecuta un comando en segundo plano cmake . && make y listo.)

6. Entorno de Pruebas (Host/DAW) -> JUCE AudioPluginHost para las pruebas, lo usare gran parte del tiempo. Ardour para demostrar que funciona en un entorno profesional.

7. Entorno de Ejecución -> VirtualBox + Ubuntu Desktop normal, para una maquina virtual mas rapida y que solo instalaremos las librerias necesarias. Unico problema: Al utilizar una maquina virtual es probable que el audio no vaya fino y se escuche algun chasquido o latencia.
